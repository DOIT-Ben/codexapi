# 项目级 Codex 工作规则

> 本文件是当前项目的主要工作规范。若与用户当前明确指令冲突，以用户当前指令为准；若与平台、安全或工具权限限制冲突，以平台和工具限制为准。无明确条款时，采取更保守的执行方式。

## 总则

- 每次回复用户之前，必须先以 `董事长，` 开头，然后再开始正式回复。
- 动手前先明确当前目标、关键假设和可验证成功标准；存在多种合理解释时先说明取舍，必要时反问。
- 任何需求增加、需求变更、架构调整或正式开发前，必须先产出规划文档并落到仓库。规划文档必须按固定路线组织：第一步用第一性原理梳理当前阶段核心需求；第二步调研该需求是否已有行业验证、成熟方法论、开源项目、skills、库、API 或最佳实践可复用；第三步结合当前项目需求说明复用、适配和必要自研的组合方式；第四步给出详细开发/落地方案、风险和验证标准。
- 保持外科式改动：只改与目标直接相关的内容，不顺手重构、格式化、改名或清理无关历史问题。
- 涉及密钥、token、账号、环境变量时，仅在任务确实需要时读取相关配置；不得在回复、日志、提交信息或文档中明文展示敏感值。
- 前端用户可见错误必须产品化表达，禁止直接暴露英文异常、堆栈、状态码或代码词。

## 项目概况

- 项目名称：codexapi / Sub2API 定制与部署工作区
- 项目类型：AI API 网关、订阅额度分发、管理后台与部署包
- 技术栈：Go、Gin、Ent、Vue 3、Vite、TailwindCSS、PostgreSQL、Redis、Docker Compose、Caddy
- 主要入口：
  - `sub2api\backend\cmd\server`：后端服务入口
  - `sub2api\frontend`：Vue 管理前端
  - `sub2api\deploy`：原项目部署配置
  - `gateway-deploy`：小规模运营部署包
- 主要输出物：可部署的 Sub2API 网关服务、管理后台、Docker Compose 部署包、运维文档

## 工作模式

默认轻量模式。

满足以下任一情况时切换严格模式：

- 预计改动 ≥ 5 个文件，或涉及跨模块架构调整。
- 单文件改动 ≥ 150 行，且影响核心逻辑。
- 涉及生产环境、权限、密钥、数据库迁移、外部写入或批量处理。
- 涉及正式交付物。
- 涉及多阶段任务，且无法在一次最小闭环内完成。

## 规划文档门禁

- 规划文档优先存放在与主题对应的 `docs\` 子目录；跨目录部署事项优先放在 `gateway-deploy\docs` 或 `sub2api\docs`。
- 只有紧急止血、小范围低风险修复、纯解释问答可以不新增规划文档。
- 规划通过后再进入实现计划和代码修改。

## 运行与验证

- 后端开发：
  ```powershell
  cd sub2api\backend
  go run .\cmd\server
  ```
- 前端开发：
  ```powershell
  cd sub2api\frontend
  pnpm run dev
  ```
- 前端构建：
  ```powershell
  cd sub2api\frontend
  pnpm run build
  ```
- 后端测试：
  ```powershell
  cd sub2api\backend
  go test .\...
  ```
- 本地部署预览优先使用 `gateway-deploy\scripts\verify.ps1`；涉及真实部署时必须核验 HTTP 可达性、容器状态和日志。

## 关键风险

- `sub2api\deploy\.env`、数据目录和运行日志可能包含敏感信息，默认不要输出全文。
- 这是 API 中转/网关类项目，生产变更涉及上游账号、用户余额、API Key、计费和限流，必须先明确回退方式。
- `gateway-deploy` 面向小规模长期运营，默认人工充值；若改为在线支付或公网生产部署，必须进入严格模式。
- 删除数据目录、数据库卷、Redis 数据或备份目录属于高风险操作，必须单独确认。

## 沟通

- 只用中文回复。
- 复杂任务按「方案 → 风险 → 验证 → 下一步」输出。
- 路径展示统一使用 Windows 风格 `\`，命令和代码遵循实际环境。
